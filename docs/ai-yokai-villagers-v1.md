# 《百鬼夜行录》AI 妖怪村民系统 — 方案草稿 v1

> 基于 Wow 1800 AI 私服模式分析 (2026-07-05)
> 关联文档：`docs/wow-1800-ai-analysis.md`

---

## 一、设计目标

让《百鬼夜行录》的世界感觉像**一个真的有人生活的妖怪世界**，而玩家是唯一的闯入者/观察者。

核心原则：
- AI 是**氛围**，不是玩法本体（不抢玩家驱动力）
- 成本可控（借鉴 DeepSeek 节流模式）
- 世界观一致（日本妖怪风，对话有时代/文化厚重感）

---

## 二、三层 NPC 模型

借鉴 WoW 私服的 "行为层 + 聊天层" 分层架构，往上再加一层玩法触发：

```
┌─────────────────────────────────┐
│  Layer 3: 玩法触发               │
│  妖怪给出试炼/悬赏/线索/交易       │
│  玩家决策影响后续事件              │
│  (轻量 LLM 调用，单次 200tokens)  │
├─────────────────────────────────┤
│  Layer 2: 对话/状态输出           │
│  LLM 生成妖怪台词、日常行为描述     │
│  世界观一致、人格化                 │
│  (仅玩家在感知范围内激活)           │
├─────────────────────────────────┤
│  Layer 1: 行为逻辑               │
│  寻路/游荡/交互/动画循环           │
│  纯 TS 状态机，零 API 成本         │
│  支持上百个并行的轻量级妖怪          │
└─────────────────────────────────┘
```

### Layer 1 — 行为层（0 API 成本）

每个妖怪最基本的生命周期：

| 行为 | 实现 | 成本 |
|------|------|------|
| 游荡 | Three.js 路径点寻路 | 0 |
| 闲站/张望 | 动画状态机 StateMachine | 0 |
| 摆摊手势 | 动画循环 | 0 |
| 简单交互反馈 | 预设动画 + 音效 | 0 |
| 跟随玩家 | 寻路+间距保持 | 0 |

**规模：** 视口内 20-30 个妖怪同时活跃无压力（参考 Three.js 粒子系统级别性能）

### Layer 2 — 对话层（LLM 按需激活）

**激活规则（节流）：**
- 仅玩家进入妖怪的 **感知半径**（比如 8 个地图格子）时激活 LLM
- 妖怪进入 "注意力状态" → 发起随机对话或继续做自己的事
- 玩家离开感知范围 → 妖怪退回 Layer 1 静默状态
- 同区域多个妖怪 → 共享一次上下文，批量生成短对话

**对话格式：**
```json
{
  "妖怪ID": "kitsune_03",
  "种类": "狐妖",
  "场景": "黄昏·妖怪集市",
  "行为": "蹲在路边摆弄一串铃铛",
  "倾向": "贪玩/好奇/有点狡诈",
  "台词": "[她抬起头，铃铛晃出细碎响声]『人类……? 好久没见到活的了。你是来买东西的，还是……来被买的?』"
}
```

**世界观约束提示词：**
> 你是日本平安时代的妖怪（具体种类）。你生活在一个人类几乎绝迹的妖怪世界里。你是[性格倾向]。你会说带有时代感的日语腔中文（简称"妖语"），语气符合你的种族和身份。不要提现代事物。对话简短，像擦肩而过的过客。

### Layer 3 — 玩法触发层（定向 LLM 调用）

当玩家主动与特定妖怪互动时：
- **式神试炼**：狐妖 → 幻术考验 / 天狗 → 速度对决
- **情报交易**：鸦天狗 → 告知隐藏宝箱位置
- **集市交易**：座敷童子 → 用某物交换稀有道具
- **支线触发**：某个妖怪在特定时间段给出特殊任务

**单次 token 消耗估计：** 200-400 tokens/次互动

## 三、成本模型（给《百鬼》的量身版）

假设：
- 玩家平均在线 2 小时/次
- 玩家周围始终有 4-8 个活跃妖怪
- 每个妖怪每 3-5 分钟触发 1 次对话
- 每 15 分钟触发 1 次玩法互动

| 场景 | 月活跃玩家 | 月调用量 | 月成本(DeepSeek) | 月成本(GPT-4o-mini) |
|------|-----------|---------|-----------------|-------------------|
| 单人单机游玩 | 1 | ~3万 tokens | **< 1元** | ~5元 |
| 小规模测试 | 50 | ~150万 tokens | **~15元** | ~300元 |
| 100 DAU | 100 | ~300万 tokens | **~30元** | ~600元 |

**结论：** DeepSeek 下成本几乎可以忽略不计。

## 五、技术落地路径

### Phase 1 — 基础框架（2-3天）
- [ ] Layer 1 行为状态机（StateMachine + 动画）
- [ ] 6 种初始妖怪模型/动画 + 路径点系统
- [ ] 感知半径检测（玩家 vs NPC 距离）

### Phase 2 — AI 接入（1-2天）
- [ ] DeepSeek API 桥接（参考 mod-ollama-chat 的 prompt 构造方式）
- [ ] 节流逻辑：仅感知范围内激活 LLM
- [ ] 妖怪人格配置文件加载
- [ ] 对话 UI 气泡/浮窗

### Phase 3 — 玩法整合（3-5天）
- [ ] 妖怪→玩家互动事件系统
- [ ] 主线剧情绑定（特定妖怪触发关键任务）
- [ ] 好感度系统（玩家行为影响妖怪态度）
- [ ] 随机事件池

## 六、与现有《百鬼》架构的对接点

- **Three.js 场景**：妖怪作为 NPC Entity 加载到场景图
- **TS 状态机**：直接复用（或扩展）现有 Entity 的行为控制
- **API 层**：新加 `ai-npc.ts` 模块，封装 LLM 调用逻辑
- **事件系统**：妖怪对话/互动事件通过现有 EventBus 分发
- **存档系统**：妖怪状态（位置、好感度、已完成互动）序列化到存档

## 七、风险与边界

| 风险 | 概率 | 缓解 |
|------|------|------|
| LLM 响应延迟（1-3s）影响沉浸感 | 中 | 预生成+缓存常见回复，LLM 仅用于"新鲜"场景 |
| 妖怪对话重复/出戏 | 中 | 人格配置 + 上下文记忆 + 多样性控制 |
| 成本失控（多人同时在线） | 低 | 严格节流：仅感知范围内激活，单次 max_tokens=150 |
| 世界观坍缩（妖怪说现代汉语） | 中 | System Prompt 硬约束 + 回复后验证过滤 |
| Three.js 性能（数十个 AI 同屏） | 低 | LOD 系统：远处妖怪降级为 sprite/2D 行走图 |
